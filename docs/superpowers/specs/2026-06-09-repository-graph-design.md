# Repository Graph Scan Artifact Design

Date: 2026-06-09
Status: Design ready for user review

## Decision

Pullwise will add a first-class repository graph artifact to every worker-backed
scan. The first version is a deterministic static repository graph, not a full
semantic RAG index. It serves two product goals:

- Users can inspect a visual map of their repository structure during and after
  a scan.
- Worker review providers receive a compact architecture summary before they
  run, improving their project-level context for issue discovery.

The graph is generated by the worker after checkout and before the LLM review.
The worker uploads it immediately through progress so the user can see structure
before the review finishes. The worker also includes it in the final scan result
for durable reconciliation. The server sanitizes and exposes it as
`repositoryGraph` in scan payloads and audit bundles. The web app renders it on
the scan screen with Cytoscape.js.

## Current System Context

The existing scan path already has the right artifact model:

- `pullwise-worker` runs `Worker.run_job`, which performs clone, preflight,
  verifier, LLM review, result upload, and checkout cleanup.
- `pullwise-server` receives worker results through
  `handle_worker_job_result`, applies them with `apply_worker_job_result`, and
  merges public scan state in `apply_worker_job_result_to_state_locked`.
- `scan_payload` already exposes nested scan artifacts such as `preflight`,
  `auditSwarm`, `verificationAudit`, and `aiUsage`.
- `pullwise-web` normalizes scan payloads in `pullwise-data.js` and renders the
  active scan in `ScanningScreen`, including preflight and audit evidence.

The repository graph should follow the same pattern as `preflight`: generated by
the worker, sanitized by the server, normalized by the web, and attached to the
scan rather than created as an unrelated endpoint.

## Goals

- Generate a repository graph for a checked-out repository without executing
  project code.
- Include enough structure to help a developer understand the repository at a
  glance: modules, entrypoints, tests, config, CI, manifests, package managers,
  language mix, and high-level dependency edges.
- Produce a compact `architectureSummary` that can be injected into Codex
  review prompts.
- Persist and expose the graph as part of scan state.
- Include the graph in JSON audit bundles and zip audit bundles.
- Render the graph interactively in the web scan experience with pan, zoom,
  node selection, and type filters.
- Keep scans resilient: graph generation failures must not block review.
- Avoid leaking secrets, local checkout paths, raw source contents, or overly
  large dependency lists.

## Non-Goals

- Building a full vector RAG store in the first version.
- Building a full language-server-grade cross-language call graph.
- Running project build/test commands to infer architecture.
- Storing raw source code snippets in `repositoryGraph`.
- Adding graph editing or user-authored annotations.
- Creating a separate graph service or graph database.
- Visualizing multiple repositories in one graph.

## Data Contract

The public scan payload includes an optional `repositoryGraph` object.

```json
{
  "version": "repository-graph/0.1",
  "generatedAt": 1710000000,
  "repo": "owner/repo",
  "branch": "main",
  "commit": "abc123",
  "summary": "React/Vite frontend with Python worker and server modules.",
  "stats": {
    "nodes": 42,
    "edges": 57,
    "files": 180,
    "directories": 24,
    "languages": ["JavaScript", "Python"],
    "entrypoints": 4,
    "tests": 12,
    "configs": 8,
    "truncated": false
  },
  "nodes": [
    {
      "id": "dir:src/screens",
      "label": "src/screens",
      "type": "module",
      "path": "src/screens",
      "size": 12,
      "importance": 0.8,
      "tags": ["frontend"]
    }
  ],
  "edges": [
    {
      "id": "file:src/App.jsx->file:src/screens/flow.jsx",
      "source": "file:src/App.jsx",
      "target": "file:src/screens/flow.jsx",
      "type": "imports",
      "weight": 1
    }
  ],
  "architectureSummary": {
    "entrypoints": ["src/App.jsx", "pullwise_worker/main.py"],
    "modules": ["src/screens", "src/lib", "pullwise_worker"],
    "reviewHints": [
      "Review scan lifecycle from frontend scan state to worker result handling.",
      "Frontend scan behavior depends on normalized scan payload fields."
    ],
    "promptText": "Repository architecture: src/App.jsx routes into src/screens, src/lib owns API state normalization, and pullwise_worker/main.py owns the worker scan lifecycle."
  }
}
```

### Node Fields

- `id`: stable graph id, prefixed by kind, such as `dir:src/screens` or
  `file:src/App.jsx`.
- `label`: short display label.
- `type`: one of `root`, `module`, `file`, `entrypoint`, `test`, `config`,
  `manifest`, `ci`, `dependency`, or `language`.
- `path`: repository-relative path when the node maps to a repo object.
- `size`: approximate weight, such as file count for modules or import count for
  files.
- `importance`: normalized 0-1 score used for layout and visual emphasis.
- `tags`: sanitized short labels such as `frontend`, `backend`, `worker`, `test`,
  `python`, or `javascript`.

### Edge Fields

- `id`: stable id derived from source, target, and type.
- `source`: node id.
- `target`: node id.
- `type`: one of `contains`, `imports`, `depends_on`, `tests`, `configures`,
  `builds`, or `uses_language`.
- `weight`: positive integer used for visual strength.

### Size Caps

The worker applies deterministic caps before upload:

- At most 120 nodes.
- At most 240 edges.
- At most 30 entrypoints.
- At most 50 dependency nodes.
- At most 20 review hints.
- At most 2 KB for `architectureSummary.promptText`.

When a repository exceeds caps, the worker aggregates low-importance file nodes
into directory module nodes and sets `stats.truncated` to `true`.

## Worker Design

Add repository graph generation to `pullwise-worker` as a static analysis unit
named `build_repository_graph(config, job, checkout_dir, preflight)`.

The function scans repository-relative paths only. It ignores directories and
files that should not influence review context:

- `.git`
- `node_modules`
- `.venv`, `venv`, `env`
- `dist`, `build`, `coverage`
- `.next`, `.nuxt`, `.cache`
- binary files and files above a small text-inspection limit

The graph builder uses lightweight deterministic parsers:

- Manifest detection from `package.json`, lockfiles, `pyproject.toml`,
  `requirements*.txt`, `Dockerfile`, `docker-compose*.yml`, GitHub workflow
  files, and common frontend/backend config files.
- Language detection from extensions and manifest context.
- Entrypoint detection from Vite/React files, package scripts, Python module
  entrypoints, CLI script declarations, server app files, worker main files, and
  GitHub workflow commands.
- Test detection from common test file names and test directories.
- Import edge extraction for Python and JavaScript/TypeScript using conservative
  static text patterns. The first version maps imports to local files when the
  target can be resolved cheaply; unresolved external imports become dependency
  nodes only when they are useful.
- Module aggregation by top-level and second-level directories.

`Worker.run_job` integrates the graph after `collect_preflight_metadata` and
before `run_codex_review`:

1. Clone repository and enforce repository limits.
2. Collect static preflight metadata.
3. Build `repository_graph`.
4. Attach `job["repository_graph"]` and
   `job["architecture_summary"]` before provider review.
5. Upload a progress update that includes `repository_graph` while the scan is
   still in the `index` phase.
6. Run verifier and LLM review.
7. Upload final result payload with the same `repository_graph`.

Graph generation failure is non-fatal. The worker records a sanitized limitation
in preflight and proceeds to verifier and LLM review. The final payload omits
`repository_graph` if no valid graph can be built.

## Agent Prompt Design

`review_prompt(job)` reads the compact graph summary when present. It does not
include the full graph. The prompt adds a bounded architecture context block:

```text
Repository architecture context:
- Entrypoints: src/App.jsx, pullwise_worker/main.py
- Main modules: src/screens, src/lib, pullwise_worker, pullwise_server
- Important edges: src/App.jsx imports src/screens/flow.jsx; Worker.run_job uploads scan result to server.
- Review hints: prioritize scan lifecycle, payload normalization, and worker result reconciliation.
Use this context to prioritize issue discovery, but report only concrete,
verifiable issues with repository-relative locations.
```

The prompt keeps existing Audit Swarm rules intact:

- Output language rules still apply.
- Existing convergence behavior still applies.
- JSON schema requirements remain unchanged.
- The architecture summary is context, not evidence. Findings still require
  concrete file locations, evidence, reproduction ideas, and verification
  results.

## Server Design

Add a public normalization helper in `pullwise-server` named
`public_repository_graph(value)`.

The sanitizer must:

- Accept both `repository_graph` and `repositoryGraph` worker result fields.
- Preserve only the public data contract fields.
- Strip absolute paths, newlines in scalar ids/labels, raw content, command
  output, and unknown nested objects.
- Cap lists again on the server to protect API responses even if a worker is
  old or misconfigured.
- Normalize counts, timestamps, tags, node types, edge types, and importance.

`apply_worker_job_result_to_state_locked` reads the sanitized graph and stores it
on the scan as `repositoryGraph` when present.

`handle_worker_job_progress` also accepts `repository_graph` or
`repositoryGraph`, sanitizes it with the same helper, and updates the running
scan immediately. This lets the polling web client render the graph before the
LLM review finishes. Final result ingestion remains the durable reconciliation
path, so losing a progress response does not lose the graph.

`scan_payload` includes:

```python
repository_graph = public_repository_graph(scan.get("repositoryGraph") or scan.get("repository_graph"))
if repository_graph:
    payload["repositoryGraph"] = repository_graph
```

`scan_audit_bundle_payload` includes `repositoryGraph` in the JSON response.
`scan_audit_bundle_zip_bytes` writes an additional `repository-graph.json` file
when the graph exists.

Result idempotency remains unchanged. The existing worker result checksum covers
the graph in the uploaded payload, and duplicate attempts keep current conflict
behavior.

## Web Design

Add graph normalization in `pullwise-web/src/lib/pullwise-data.js` through
`normalizeRepositoryGraph(value)`.

`normalizeScan` attaches `repositoryGraph`.

Add a reusable scan graph panel component named `RepositoryGraphPanel`.

The panel is rendered in `ScanningScreen` near preflight and audit evidence for
single-repository scans. Batch scans can initially show a per-scan graph only
when the active scan is selected by existing scan state; a combined batch graph
is out of scope.

The UI uses Cytoscape.js because the feature is a network/dependency graph, not
a node editor. Required interactions:

- Pan and zoom.
- Fit graph.
- Toggle node types.
- Click a node to inspect path, type, tags, and connected counts.
- Highlight directly connected nodes and edges.
- Responsive layout for desktop and mobile.

The panel does not render when no graph exists. If graph generation failed, the
existing preflight/audit evidence surfaces the limitation rather than showing an
empty graph panel.

## Security And Privacy

The graph must never contain:

- Worker checkout absolute paths.
- Server filesystem paths.
- GitHub clone tokens.
- Worker tokens.
- Codex session details.
- Raw source contents.
- Environment variable values.
- Command output.
- Hidden file contents.

All paths in nodes and edges are repository-relative. Scalar text fields are
single-line, length-capped, and stripped of NUL bytes. The server sanitizer is
the final trust boundary for public API output.

## Performance

Graph generation runs in the worker process and should complete quickly for
normal repositories. It must honor existing repository file and byte limits and
apply its own inspection limits:

- Do not read files that are likely binary.
- Inspect only a capped prefix for import detection.
- Avoid recursive parsing of generated or dependency directories.
- Use deterministic aggregation before upload.
- Keep graph payload small enough for scan polling responses.

The first implementation should target a graph generation budget under a few
seconds for repositories already accepted by existing preflight limits.

## Testing Plan

### Worker Tests

- `build_repository_graph` detects manifests, languages, entrypoints, tests,
  config, CI, and local import edges from a small fixture repository.
- Graph output is deterministic.
- Graph output contains only repository-relative paths.
- Graph caps aggregate oversized repositories and set `stats.truncated`.
- Graph generation failure does not fail `run_job`.
- `run_job` uploads `repository_graph` through progress before provider review.
- `run_job` uploads `repository_graph` in successful payloads.
- `review_prompt` includes the architecture summary when present and omits it
  when absent.
- The architecture prompt block stays within the configured size cap.

### Server Tests

- Worker result with `repository_graph` persists sanitized `repositoryGraph` on
  the scan.
- Worker progress with `repository_graph` updates a running scan's
  `repositoryGraph`.
- `scan_payload` exposes `repositoryGraph`.
- Unsafe graph fields are removed or normalized.
- Oversized worker graph arrays are capped by the server.
- Audit bundle JSON includes `repositoryGraph`.
- Audit bundle zip includes `repository-graph.json` when present.
- Missing or invalid graphs do not break result ingestion.

### Web Tests

- `normalizeScan` normalizes `repositoryGraph`.
- `ScanningScreen` renders the graph panel when a completed scan includes graph
  data.
- The graph panel does not render for missing graph data.
- Type filters hide and show nodes.
- Clicking a node shows node details.
- Existing scan progress, preflight evidence, audit evidence, and bundle
  download behavior continue to render.

## Rollout

The rollout can be incremental:

1. Add worker graph builder and prompt summary injection.
2. Add server sanitizer, scan payload exposure, and audit bundle inclusion.
3. Add web normalization and graph panel.
4. Add Cytoscape.js dependency and build verification.
5. Verify with unit tests for worker, server, and web, then run each package's
   normal build/test command.

Older workers that do not send `repository_graph` remain compatible because the
field is optional. Older web builds remain compatible because server payloads add
an optional field without changing existing fields.

## Acceptance Criteria

- A scan of a representative repository produces a non-empty `repositoryGraph`
  with nodes, edges, stats, and architecture summary.
- Web polling can receive and display `repositoryGraph` before the LLM review
  completes.
- The same scan's review prompt includes the graph summary before provider
  execution.
- Server scan payload includes sanitized `repositoryGraph`.
- Audit bundle JSON and zip include the graph.
- Web scan screen displays the graph interactively.
- Graph generation failures do not fail scans.
- Public payloads do not expose secrets, absolute paths, raw source content, or
  command output.
- Worker, server, and web tests cover the new behavior.
